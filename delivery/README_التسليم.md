## ٢. الفكرة الجوهرية (اقرأها أولاً)

**المخطط ليس بيانات — هو لوحة خلفية.** كل «غرض» (موظفون، طفايات، كاميرات، مجموعات) هو **طبقة نقاط** تُرسم فوق نفس مخطط الطابق. لا تحتاج مخططاً لكل غرض؛ **مخطط واحد لكل طابق** + جدول نقاط.

**كل نقطة = كود موقع** موجود أصلاً في النظام. النقطة تحمل إحداثي `(x, y)` نسبي فقط؛ حالتها الحيّة تأتي من سجلّات النظام المربوطة بنفس `locationId`.

---

## ٣. ما هو موجود فعلاً في الباك-إند (لا تُعِد بناءه)

من مراجعة `app.js` الحالي:

| موجود | الوصف |
|------|------|
| `locationOperationalStatus(locationId)` | يجمّع لكل موقع: آخر تنظيف، استحقاق التنظيف، البلاغات المفتوحة، طلبات الضيافة، تجاوزات SLA. **هذا مصدر تلوين النقطة.** |
| `/api/facilities/heatmap` | يُرجع لكل مساحة `heat_score` (٠–١٠٠) و`level` و`reasons[]`. |
| الهيكل الهرمي | مرفق ← مبنى ← طابق ← منطقة ← مساحة، عبر `/api/facilities/*`. |
| `space.legacy_location_id` | يربط المساحة الجديدة بكود الموقع القديم (مثل `MF-WS-03`). |
| `tickets` / `reports` / `hospitalityOrders` | كلها تحمل `locationId`، والتذاكر تحمل `module` (`cleaning`/`maintenance`/`hospitality`). |
| شاشة `adminMaps()` | **فارغة حالياً** (شاشة «قريباً») — هي المكان الطبيعي لزرع الخريطة. |

**الخلاصة:** الفجوة الوحيدة هي **إحداثيات النقاط** + **واجهة العرض**. الحالة جاهزة.

## ٤. ما يجب إضافته (الحد الأدنى)

### ٤.١ جدول إحداثيات
```sql
CREATE TABLE map_points (
  id           INTEGER PRIMARY KEY,
  floor        TEXT NOT NULL,        -- 'MF','GF','1F'...
  code         TEXT NOT NULL,        -- كود الموقع: 'MF-WS-03'
  x            REAL NOT NULL,        -- نسبة مئوية 0..100 على عرض المخطط
  y            REAL NOT NULL,        -- نسبة مئوية 0..100 على ارتفاع المخطط
  layer        TEXT NOT NULL,        -- 'cleaning'|'safety'|'cameras'|'groups'|'hospitality'
  type         TEXT,                 -- مشتق من الكود: WS,M,MR,BR,FS,CAM...
  UNIQUE(floor, code, layer)
);
```
> الإحداثي نسبي (٪) كي يصمد مع أي حجم عرض. الزاوية المرجعية: أعلى-يسار المخطط = (0,0).

### ٤.٢ نقاط نهاية
```
GET  /api/maps/floors                 → قائمة الطوابق + رابط SVG لكل طابق
GET  /api/maps/:floor/points          → نقاط الطابق [{code,x,y,layer,type}]
PUT  /api/maps/:floor/points          → حفظ/تحديث النقاط (من محرّر الإحداثيات)
GET  /api/maps/:floor/status?layers=cleaning,maintenance,...
     → يدمج النقاط مع locationOperationalStatus() لكل كود، ويُرجع الحالة الحيّة
```
> الأخيرة هي القلب: تأخذ نقاط الطابق، وتنادي `locationOperationalStatus()` (أو `/facilities/heatmap`) لكل `code`، وتُرجع لكل نقطة لونها/حالتها لكل قسم مُفعّل.

### ٤.٣ تخزين المخططات
ضع ملفات `floors/*.svg` كأصول ثابتة (static). كلها `viewBox="0 0 1191 842"` — نظام إحداثيات موحّد لكل الطوابق.

## ٥. تدفّق العمل (خطوة بخطوة)

**أ. مرة واحدة لكل طابق — وضع النقاط:**
1. افتح `MRFQ Coordinate Editor` (أو انقل منطقه لشاشة admin).
2. اختر الطابق → اختر كوداً من القائمة → انقر موضعه على المخطط.
3. يُحفظ `(x,y)` تلقائياً. كرّر حتى تكتمل النقاط.
4. «تصدير JSON» → ادفعه إلى `PUT /api/maps/:floor/points`.

**ب. التشغيل اليومي — العرض الإداري:**
1. المستخدم يفتح شاشة «الخرائط».
2. الواجهة تجلب `GET /api/maps/MF/status?layers=...`.
3. ترسم النقاط فوق SVG، ملوّنة حسب أسوأ حالة بين أقسامها المُفعّلة.
4. مفاتيح الطبقات تُظهر/تخفي كل قسم (ترشيح حسب `layer`/`module`).
5. النقر على نقطة يفتح بطاقة الموقع (كل الأقسام المرتبطة بنفس `locationId`).

**ج. التحديث الحيّ:** عند ورود تقرير/بلاغ/طلب، ادفع تحديثاً عبر SSE (النظام يستخدم بثّاً مباشراً أصلاً) فيتغيّر لون النقطة فوراً.

## ٦. ربط الأقسام (المنطق الفعلي)

نقطة واحدة `MF-WS-03` قد تحمل ثلاث حالات معاً — لأنها سجلّات منفصلة تتشارك `locationId`:

```js
// لكل code في نقاط الطابق:
const id = locationIdOf(code);                  // عبر legacy_location_id
const cleaning   = lastReport(id, 'cleaning');  // حالة النظافة
const maintenance= openTickets(id,'maintenance');// بلاغات الصيانة
const hospitality= openHospOrders(id);          // طلبات الضيافة
const safety     = assetStatus(code);           // طفايات (طبقة أصول)
// لون النقطة = أعلى خطورة بين الطبقات المُفعّلة
```
- **طبقة = ترشيح حسب `module`/`layer`.** لا تعارُض: الموقع لا «يملكه» قسم.
- طبقتا «السلامة» و«الكاميرات» تعتمدان جدول أصول (`assets`) بكود ونوع وحالة فحص — أضفهما لاحقاً كطبقة نقاط بنفس النمط.

## ٧. ملاحظات تدقيق البيانات
- راجع `data/duplicates_report.md` قبل الاستيراد: **٤٢ كوداً مكرراً**، أغلبها بسبب ترقيم الطابق الخامس كـ `6F` بالخطأ.
- اجعل `code` مفتاحاً فريداً (UNIQUE) وامنع الأكواد الفارغة عند الاستيراد.

---

## ٨. أولوية التنفيذ المقترحة
1. جدول `map_points` + `GET/PUT points` + استضافة SVG.
2. شاشة محرّر الإحداثيات (admin) — لملء النقاط.
3. `GET /status` بدمج `locationOperationalStatus`.
4. شاشة العرض الإداري (طبقات + بطاقة موقع) في `adminMaps()`.
5. طبقات السلامة/الكاميرات (جدول أصول) + التحديث الحيّ SSE.
